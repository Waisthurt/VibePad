# VibePad 项目说明

> 目的：这是一份给后续维护者（包括其他 AI）使用的项目总览。开始修改前先读本文件，再结合 `README.md`、`protocol/protocol.md` 与相关源码确认细节。

## 1. 项目是什么

VibePad 是一个把 Android 手机变成 Windows 电脑局域网输入终端的工具。手机可以向电脑输入长文本、发送常用快捷操作，并作为多点触控板控制鼠标。

典型使用场景是：电脑上运行 AI 编程工具、终端或编辑器，用户用手机输入提示词、复制粘贴内容、滚动网页/文档和控制鼠标。

当前稳定发布版本：**0.1.4**

## 2. 当前已实现能力

### Android 手机端

- 手动填写电脑局域网 IPv4 地址并连接
- 多行文字输入、系统语音输入后的文字发送
- 回车、退格长按、复制、粘贴、上传、截图等快捷操作
- 单指移动鼠标、轻点左键、长按拖拽
- 双指滚动、双指轻点右键
- 鼠标灵敏度设置（`0.5×` 到 `3×`）
- 双指滚动灵敏度设置（`0.5×` 到 `3×`）
- 默认开启的前台自动重连：回到 App 后连接上次成功的电脑，失败时有限退避重试
- 高频鼠标移动与双指滚动均采用固定 8ms 节奏合帧

### Windows Agent

- 系统托盘常驻、开机自动启动、单实例运行
- 面向普通用户的深色淡紫色引导窗口，自动给出推荐填写的局域网 IP
- WebSocket 服务（TCP `8765`）
- 鼠标与滚动 UDP 快速通道（UDP `8767`）
- 剪贴板、键盘、鼠标、滚轮、截图注入
- 安装器内置 .NET 运行时，自动添加局域网防火墙规则

## 3. 系统架构

```text
Android VibePad
  ├─ WebSocket：可靠控制命令、文字、按键、点击、WebSocket 兜底
  └─ UDP：高频鼠标移动与双指滚动
          │
          ▼
Windows VibePad Agent
  ├─ Kestrel WebSocket Server（8765）
  ├─ UDP Receiver（8767，仅接受已连接手机的 IP）
  ├─ 8ms 输入缓冲/合帧
  ├─ ClipboardWorker
  └─ SendInput 注入 Windows 键盘和鼠标事件
```

两端必须在同一局域网、同一 Wi-Fi 或同一手机热点中。电脑接网线、手机连同一路由器 Wi-Fi 的组合也可用。

## 4. 通信与低延迟设计

### 基础连接

- WebSocket 地址：`ws://<电脑IP>:8765/vibepad/`
- 用户在手机界面只输入 `<电脑IP>`，例如 `192.168.1.3`
- 不需要输入 `ws://`、端口号或路径

### 高频输入链路

触控事件可能会成批到达；如果每次触摸事件都直接经 TCP 发送，Wi-Fi 调度与 TCP 队列会直接表现为光标抖动。因此当前方案是：

1. Android 先累积位移与滚动量
2. 每 **8ms（约 125Hz）** 合并发送一次
3. 优先通过 UDP `8767` 发送
4. Windows 端再次用 8ms 缓冲合并，再调用 `SendInput`
5. 同时保留 WebSocket 兜底，UDP 被防火墙阻断时仍可使用

UDP 数据包定义：

| 类型 | 格式 |
| --- | --- |
| 鼠标移动 | 8 字节，小端 `float32 dx` + `float32 dy` |
| 双指滚动 | 5 字节，`0x01` + 小端 `int32 delta` |

Windows Agent 连接后会返回：

```json
{"type":"udp_ready","port":8767,"scroll":true}
```

只有收到 `scroll: true` 的新版 Android 才启用 UDP 滚动。旧手机端或旧电脑端会自动继续使用 WebSocket，所以协议保持向后兼容。

注意：网页本身可能实现惯性/平滑滚动，因此不同网页的主观顺滑度会有差异；UDP 合帧的目标是减少输入链路自身的节奏抖动，而不是替网页实现动画。

## 6. 开发、构建与安装

### Windows Agent

本地构建：

```powershell
$dotnet = 'C:\Program Files\dotnet\dotnet.exe'
& $dotnet build .\windows\VibePad.Agent\VibePad.Agent.csproj -c Release
```

生成自包含安装器：

```powershell
.\installer\build-installer.ps1 -Version 0.1.3
```

产物位于：`installer/output/VibePad-Agent-Setup-<版本>.exe`

### Android

当前 Android 版本位于 `android/app/build.gradle.kts`。本机 Android Studio 环境通常需要：

```powershell
$env:JAVA_HOME = 'C:\Program Files\Android\Android Studio\jbr'
$env:ANDROID_HOME = 'C:\Users\caiye\AppData\Local\Android\Sdk'
$env:Path = "$env:JAVA_HOME\bin;$env:ANDROID_HOME\platform-tools;$env:Path"
cd .\android
.\gradlew.bat :app:assembleDebug
```

APK 位于：`android/app/build/outputs/apk/debug/app-debug.apk`

### 非常重要：源码、发布目录与已安装程序是三份不同的东西

用户日常从托盘或开始菜单打开的是：

```text
C:\Program Files\VibePad Agent\VibePad.Agent.exe
```

只修改 `D:\vibepad` 源码或只执行 `dotnet build`，**不会**更新这个已安装副本。改完 Windows Agent 后若要让用户看到效果，必须：

1. 重新运行 `build-installer.ps1`
2. 用生成的安装器覆盖安装（需要管理员权限）
3. 重启 Agent

这曾经多次导致“源码是新版、托盘打开却仍是旧版”的误判。排查时优先检查运行进程路径与 `C:\Program Files\VibePad Agent\VibePad.Agent.exe` 的修改时间/哈希。

## 7. 发布与分享流程

推荐每个稳定版本按以下步骤执行：

1. 增加 Windows 安装器版本：`installer/VibePad-Agent.iss` 与 `installer/build-installer.ps1`
2. 若 Android 有用户可见或协议变动，同时增加 `versionCode` 与 `versionName`
3. 构建 Windows 安装器和 Android APK
4. 在真实 Windows + Android 局域网环境测试
5. 提交并推送 GitHub
6. 建立分享目录，包含：
   - `VibePad-Agent-Setup-<版本>.exe`
   - `VibePad-Android-<版本>.apk`
   - `使用说明.txt`
7. 压缩成 `VibePad-<版本>-Windows-Android.zip`

当前 GitHub 仓库：<https://github.com/Waisthurt/VibePad>

## 8. 最小回归测试清单

- Windows 安装器能在干净的 Windows 11 上完成安装并启动 Agent
- Agent 窗口只突出显示一个推荐 IP，手机用户不需要判断 `ws://` 地址
- 手机与电脑同网时可以连接、断开后可以一键重连
- 文本发送、回车、退格、复制、粘贴、截图均正常
- 单指移动、左键、右键、长按拖拽正常
- 双指滚动方向正确，普通网页和滚动较卡的网页都应测试
- UDP 被禁用/防火墙阻断时，鼠标与滚动仍通过 WebSocket 可用
- 手机切到后台后允许断线；回到 App 后应自动重连上次成功的电脑

## 9. 安全与兼容性边界

- 当前**没有安全配对、设备令牌或身份认证**
- 只应在可信任局域网中使用，绝不能暴露端口 `8765` 到互联网
- 安装器防火墙规则限制为 `localsubnet`，但这不是身份认证的替代品
- 目标组合为 Windows 11 + Android（最低 Android API 26）
- 分享的 Android APK 当前为 debug 签名；若准备公开分发，应建立正式签名密钥、发布签名 APK 并规划更新机制

## 10. 后续建议优先级

1. 安全配对：一次性六码配对 + 已授权设备令牌
2. 自动发现：局域网发现电脑，减少手动输入 IP
3. 正式 Android 签名与发布渠道
4. 版本更新机制：安装器升级检测、Android 更新提示
5. 输入性能实测：采集鼠标/滚动延迟与丢包指标，再决定是否继续优化采样或 UDP 策略

## 11. 修改时的原则

- 高频移动/滚动不要退回“每个触控事件一个 WebSocket 消息”的实现
- 任何协议扩展都必须保留旧客户端/旧 Agent 的回退路径
- 改 Windows 窗口布局时优先使用自适应布局，避免固定子容器高度造成文字裁切
- 面向普通用户的文案应直接说明“手机该填什么”，避免暴露协议 URL
- 发布前务必验证“实际运行的 Program Files 副本”而不是只看源码编译结果
